# Домовой vs существующие решения

Домовой не конкурирует с каждым инструментом в отдельности — он конкурирует с самой идеей держать десять разных приложений, данные которых не связаны между собой.

## Ключевые отличия

| | Домовой | Anytype | Jira | Dynamics AX | Bitrix24 |
|--|---------|---------|------|------------|---------|
| Модель данных | Граф (`thing` + 25 рёбер) | Объекты + типы | Issues + Projects | ERP-модули | Сущности CRM |
| Связи между доменами | Нативные типизированные рёбра | Отношения между типами | Только issue links | Интеграция модулей | Слабые |
| Self-hosted | Полностью | Частично¹ | Да (дорого) | On-premise (лицензия) | Да (коробка) |
| Запросы к данным | SurrealQL (граф-обход) | Фильтры | JQL | AX Query | Нет |
| Автоматизация | Плейбуки (встроено) | Нет | Automation for Jira | Workflow engine | Роботы / триггеры |
| AI-интеграция | RAG, агенты, мульти-модель | Ограничена | Atlassian Intelligence | Copilot (MS) | Базовая |
| Производство / маршруты | Плейбуки + операции | Нет | Нет | Да (полноценный) | Нет |
| Финансовый учёт | Базовый (через datasource) | Нет | Нет | Да (полноценный) | Базовый |
| IT-инфраструктура | Встроена | Нет | Частично | Нет | Нет |
| Обучение / LMS | Встроено | Нет | Нет | Нет | Нет |
| Мультиязычность | `_lang` + `_i18n` | Нет | Да | Да | Да |
| Федерация | Да (HLC) | Нет | Нет | Нет | Нет |
| Стоимость | Бесплатно (self-hosted) | Бесплатно | От $8/user/мес | Сотни тыс. ₽/год | От 0 до дорого |

¹ Anytype синхронизирует через собственные p2p-серверы; fully self-hosted нестандартен.

## Унифицированные рабочие пространства

Наиболее близкие конкуренты — инструменты с объектной или граф-моделью данных. Именно с ними Домовой пересекается концептуально сильнее всего.

| Инструмент | Модель данных | Self-hosted | Автоматизация | AI | Федерация | Домовой |
|-----------|--------------|------------|--------------|-----|----------|---------|
| **Anytype** | Объекты + типы + отношения | Частично¹ | Нет | Ограничена | Нет | Граф глубже, плейбуки, IT-слой |
| **AppFlowy** | Блоки + базы данных | Да | Нет | Плагины | Нет | Граф нативный, автоматизация |
| **Affine** | Блоки + whiteboard | Да | Нет | Базовая | Нет | Структурированный граф vs canvas |
| **Capacities** | Объекты + типы | Нет (облако) | Нет | Базовая | Нет | Self-hosted, SurrealQL |
| **Fibery** | Гибкие схемы + отношения | Платно | Базовая | Плагины | Нет | Граф-обход, self-hosted |
| **Tana** | Суперноды + теги | Нет | Нет | AI-native | Нет | Self-hosted, федерация |
| **Heptabase** | Карточки + whiteboard | Частично | Нет | Базовая | Нет | SurrealQL, автоматизация |
| **Coda** | Docs + таблицы + формулы | Нет | Базовая | Плагины | Нет | Self-hosted, граф |

¹ Anytype синхронизирует через собственные серверы (p2p-протокол), self-hosted бэкенд технически возможен, но нестандартен.

### Anytype — ближайший конкурент

Anytype — наиболее концептуально близкий инструмент: объекты (≈ `thing`), типы, отношения, граф-вид, local-first, open source.

**Что Anytype делает хорошо:**
- Отличный UX для создания объектов и связей
- E2E-шифрование, локальные данные
- Граф-вид для навигации
- Открытый исходный код
- Мобильные приложения

**Чего не хватает относительно Домового:**
- Нет SurrealQL — запросы только через UI-фильтры, не произвольный граф-обход
- Нет плейбуков / автоматизации — требует n8n рядом
- Нет AI-интеграции с личными данными (RAG, агенты, tool use)
- Нет одноразовых сред для обучения
- Нет IT-инфраструктурного слоя (серверы, сети, инциденты)
- Нет федерации между инстансами
- Нет встроенного мультиязычного слоя (`_i18n`)
- Синхронизация через серверы Anytype — не полностью self-hosted
- Нет дашбордов с внешними datasource (Prometheus, InfluxDB)
- Нет голосового ассистента / Telegram-бота

**Где Anytype лучше:**
- Готов к использованию прямо сейчас
- Мобильные приложения зрелее
- UX отполирован
- Не требует администрирования сервера

**Вывод:** Anytype — отличный инструмент для PKM и заметок. Домовой — платформа, где данные связаны с IT-инфраструктурой, автоматизацией, AI-агентами и обучением в единой модели.

### AppFlowy

Open source альтернатива Notion. Хороший UX, активное сообщество, self-hosted. Но данные хранятся как документы с базами, не как граф — связи между разными «приложениями» (задачи, заметки, база данных) слабые. Нет автоматизации и AI.

### Affine

Local-first, open source, сочетает Notion-блоки с Miro-whiteboard. Хорош для визуального мышления и заметок. Граф-модели данных нет — whiteboard это canvas, а не граф отношений. Нет автоматизации.

### Fibery

Наиболее гибкий инструмент для командного управления работой: кастомные типы, отношения, формулы, базовые автоматизации. Ближе всего к «построй свой инструмент». Но облачный (self-hosted есть, но дорогой), нет граф-обхода запросами, нет AI-native.

### Tana

AI-native outliner с суперногами (суперноды = типизированные узлы). Интересная модель, но только облако, нет self-hosted, нет автоматизации и интеграций.

## По областям

### Задачи и GTD

| Инструмент | Сильные стороны | Что не умеет | Домовой |
|-----------|----------------|-------------|---------|
| **Todoist** | Простота, кроссплатформенность | Нет зависимостей между задачами, нет связи с инвентарём | `depends_on`, `about`, `part_of` — нативно |

**Преимущество Домового:** задача может быть `about` сервера, `part_of` плейбука, `produces` документа — всё в одной модели без переключения инструментов.

### Управление проектами и разработкой (Jira и аналоги)

| Инструмент | Сильные стороны | Что не умеет | Домовой |
|-----------|----------------|-------------|---------|
| **Jira** | Agile (Scrum/Kanban), JQL, огромная экосистема Atlassian | Облако / дорого, задачи изолированы от других данных | Self-hosted, задача связана с сервером, инцидентом, документом |
| **YouTrack** | Self-hosted (JetBrains), мощный, гибкий | Только трекер задач, нет связей с другими доменами | Граф нативный |
| **Plane** | Open source Jira-альтернатива, self-hosted | Молодой, нет экосистемы | То же + плейбуки |
| **Linear** | Скорость, UX, циклы вместо спринтов | Облако, только для разработки | Self-hosted, не только IT |
| **GitLab Issues** | Встроено в Git-платформу, CI/CD связь | Только разработка, нет широкого контекста | Граф шире |
| **Azure DevOps** | Полный ALM-цикл, Microsoft-экосистема | Облако / enterprise, сложность | Self-hosted |

**Jira → Домовой: маппинг:**

| Jira | Домовой |
|------|---------|
| Issue (Epic/Story/Task/Bug) | `kind: задача` с полем `issue_type` |
| Issue hierarchy | `part_of` (эпик → история → подзадача) |
| Issue links (blocks, duplicates) | `depends_on`, `related_to` |
| Sprint | `kind: спринт` → `part_of` → `kind: проект` |
| Board (Scrum/Kanban) | `kind: вид` + `kind: умный-список` |
| Workflow transitions | `kind: workflow-схема` + плейбук |
| JQL | SurrealQL (аналогичная концепция) |
| Components / Labels | `about` → `kind: компонент`, `tags` |
| Fix Version | `about` → `kind: версия` |
| Dashboards / Gadgets | `kind: дашборд` + `kind: виджет` |
| Automation for Jira | Плейбуки |
| Confluence (wiki) | Wiki-раздел в Домовом |

**Главное отличие:** задача в Jira — изолированный объект. Задача в Домовом может быть `about` реального сервера, `triggered_by` мониторингового алерта, `produces` документ, `assigned_to` человека чей контакт, инвентарь и история — тут же в графе.

### ERP-системы (Axapta / Dynamics AX, 1C, Odoo)

| Инструмент | Сильные стороны | Что не умеет | Домовой |
|-----------|----------------|-------------|---------|
| **Microsoft Dynamics AX / F&O** | Производство, маршрутизация, полный финансовый учёт | Enterprise-стоимость, сложность, не self-hosted | Маршруты и workflow через плейбуки; бухучёт — через datasource |
| **1C:ERP / УПП** | Российский рынок, бухучёт РСБУ, зарплата, глубокая интеграция | Закрытый, только Windows-экосистема, устаревший UX | Open, граф-нативный; 1С как datasource для финансов |
| **Odoo** | Self-hosted, открытый, широкий функционал (CRM+склад+бухучёт) | Тяжёлый деплой, монолит, нет граф-модели | Lighter, граф нативный; Odoo как datasource |
| **SAP** | Промышленный стандарт, всё что угодно | Цена, сложность, консультанты | Не конкурент — разные масштабы |

**Axapta → Домовой: что ложится:**

| Axapta / Dynamics | Домовой |
|-------------------|---------|
| Route (производственный маршрут) | `kind: маршрут` → `contains` → `kind: операция` (ordered via `depends_on`) |
| Route operation | `kind: операция` с work_center, setup_time, run_time |
| Work center | `kind: рабочий-центр` с capacity, cost_per_hour |
| BOM (Bill of Materials) | `requires` / `needs` рёбра с quantity, unit |
| Production order | `kind: запуск-плейбука` по маршруту |
| Workflow (согласования) | Плейбуки + `requires_approval_from` + условные шаги |
| Purchase order | `kind: заказ` + `needs` + плейбук согласования |
| Service request | `kind: инцидент` + `assigned_to` + `about` |
| Project (time & expense) | Задачи + `part_of` проект + учёт времени |

**Что не стоит тащить в Домовой:**

| Axapta | Почему | Альтернатива |
|--------|--------|-------------|
| Двойная запись, план счетов, ГК | Регуляторная специфика, аудит | 1С / Odoo как datasource |
| MRP / MPS (планирование потребностей) | Сложные алгоритмы оптимизации | Воркер-модуль с OR-библиотекой |
| Расчёт зарплаты + налоги | Региональное законодательство | 1С ЗУП через datasource |
| Полноценный WMS (ячейки, паллеты) | Специализированная область | Отдельная WMS через datasource |

Финансовый учёт подключается через `kind: источник-данных` (subtype: `odata` для Dynamics, `1c-xml` или REST для 1С) — данные доступны в графе как контекст, но не дублируются.

| **Odoo** | ERP + CRM, self-hosted, открытый | Сложность, тяжёлый деплой | Легче, граф-нативный |
| **Monica** | Personal CRM, self-hosted, напоминания | Только контакты, нет связи с задачами/финансами | Контакт — `kind: человек`, связан со всем |
| **Apple Contacts / Google** | Синхронизация, экосистема | Облако, нет связей | Self-hosted, граф |

**Bitrix24 — подробнее.** Наиболее популярный в СНГ комбайн с self-hosted опцией. Закрывает CRM, задачи, чат, диски, HR — но каждый модуль живёт в своей базе, связи между ними ограничены. Автоматизация (роботы и триггеры) слабее плейбуков. Нет AI-агентов с доступом к данным. Тяжёлый деплой и обновления. Лицензия коробки дорогая.

**Домовой для малого бизнеса:** контакт (`kind: человек`) связан с задачами (`assigned_to`), сделками (`about`), документами (`represents`), платежами, историей переписки (чаты `about` этот контакт) — всё через рёбра одного графа. Плейбук заменяет «роботов Bitrix» без отдельного конструктора.

## Для малых команд

Домовой работает для небольших внутренних команд (5–20 человек) без каких-либо изменений модели:

- **Любой с браузером** заходит через веб — Домовой это Next.js, установка не нужна
- `can_access` с permissions — тонкая настройка прав на любой узел
- Назначение задач, группы, согласования (`requires_approval_from`) — встроено
- Общие дашборды, умные списки, плейбуки — шарятся через `can_access`
- История изменений (`kind: изменение`) — аудит-лог кто что менял
- Публичные узлы через `thing:public` — клиент видит только то, что ты открыл

Для внешнего клиента или подрядчика достаточно создать аккаунт с ограниченными правами или дать ссылку на публичный граф — никакой установки.

Если у партнёра есть свой инстанс Домового — федерация через shadow namespace.

## Когда Домовой — не лучший выбор

| Ситуация | Лучше использовать |
|----------|-------------------|
| Одновременное редактирование одного текста (Google Docs-стиль) | Google Docs, Notion, Confluence |
| Простые задачи без связей | Todoist, Apple Reminders |
| Только финансы и бюджет | YNAB, Actual Budget |
| Только мониторинг инфраструктуры | Grafana + Prometheus |
| Нет никого, кто поднимет self-hosted сервер | Любой SaaS |
| Нужна корпоративная поддержка и SLA | Enterprise-инструменты |
| Полноценный бухгалтерский учёт (РСБУ, двойная запись, отчётность) | 1С:Бухгалтерия, Odoo, Dynamics |

## Итог

Домовой — это не «ещё один менеджер задач». Это инфраструктура для личных и семейных данных, где граф связей между доменами важнее, чем функциональность каждого домена по отдельности.

Ценность проявляется не сразу, а когда задача об автомобиле связана с расходами на него, напоминает о ТО, хранит документы и историю ремонтов — и всё это видно из одного места без копирования данных между приложениями.
