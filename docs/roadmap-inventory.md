# Роадмап: первая вертикаль — инвентарь и места хранения

Узкий первый домен, который **прокатывает всю архитектуру** один раз и проверяет её на живом. Дальше любой следующий домен (документы, задачи, медицина, IT) едет по тем же рельсам без переделок.

Связанные документы: [`database.md`](database.md), [`kind-audit.md`](kind-audit.md), [`relation-typing.md`](relation-typing.md), [`access-control.md`](access-control.md), [`ui-architecture.md`](ui-architecture.md).

## Зачем именно инвентарь

| Свойство | Почему хорошо как первая вертикаль |
|---|---|
| Маленький домен | 2–3 `kind` (`item`, `location`, `person`), 3–4 ребра (`contains`, `lent_to`, `can_access`, `part_of`) |
| Живой у тебя сразу | ~1000 узлов в seed уже есть, не надо ничего выдумывать |
| Прокатывает все слои | ASCII-канонизация / SKOS-словари / Wikidata-якоря / иерархия категорий / UI из каталога / i18n / права / мобильный сценарий |
| Узкий UX-набор | список / карточка / форма / дерево локаций — четыре экрана |
| Завершённость | после фазы 6 можно реально пользоваться для семейного учёта |

## В скоупе

- **`kind`**: `item`, `location`, `person` (+ мета: `vocabulary`, `concept`)
- **рёбра**: `contains`, `lent_to`, `borrowed_from`, `can_access`, `part_of`, `participant`, `identified_by`, `related_to` (теги)
- **категории**: дерево из `kind-audit.md` (~65 концептов в `vocab:domovoy_category`)
- **бренды**: `vocab:domovoy_brand` (~5 стартовых + наращивание)
- **внешние якоря**: Wikidata (категория / модель / бренд), опц. GTIN
- **UI-экраны**: `<NodeList>`, `<NodeCard>`, `<NodeForm>`, `<LocationTree>`
- **виджеты**: ~10–12 (см. фазу 3)
- **i18n**: переключатель ru/en, метки из `_i18n` концептов
- **права**: `DEFINE ACCESS RECORD` + `PERMISSIONS WHERE` для `thing` и `can_access`
- **PWA**: офлайн-список, камера для добавления, опц. сканер штрих-кода

## Вне скоупа (явно — чтобы не расползалось)

- ❌ AI/RAG, чаты, embeddings
- ❌ плейбуки и сложная автоматизация
- ❌ документы, файлы, OCR
- ❌ медицина, диагнозы, рецепты
- ❌ задачи и пайплайны (`task`/`appeal`/`incident`/`payment`)
- ❌ животные и растения (добавятся в следующей вертикали)
- ❌ детальные дашборды и графики
- ❌ Telegram-бот и голос

## Архитектурные валидации (что каждая фаза подтверждает)

| Фаза | Валидирует |
|---|---|
| 1 — миграция seed | ASCII канонизация, SKOS-словари, иерархия категорий |
| 2 — каркас web | подключение к SurrealDB, маршрутизация, auth-стаб |
| 3 — `<NodeCard>` | универсальная карточка по каталогу, контракт виджета |
| 4 — `<NodeList>` | таблица по `ui.list_columns`, фильтры по иерархии категорий |
| 5 — `<NodeForm>` | форма по `ui.form_create`, валидация рёбер из каталога |
| 6 — `<LocationTree>` | специализированный UX без нарушения архитектуры |
| 7 — i18n переключатель | симметричная многоязычность через `_i18n` концептов |
| 8 — права и шаринг | `PERMISSIONS WHERE` читает граф (по `access-control.md`) |
| 9 — `lent_to` workflow | действия-виджеты меняют граф |
| 10 — PWA и мобильный | адаптивная раскладка автоматически + офлайн-кеш |

После фазы 5 — концепция UI доказана. После фазы 8 — вся архитектура (свобода + права + i18n) прокатана. После фазы 10 — это рабочий продукт для семьи.

**Артефакт.**

### Фаза 3: `<NodeCard>` для `item`

**Цель.** Универсальная карточка узла, рендерится из каталога. Сразу для `item` (потом любой `kind` подцепится через декларацию).

**Артефакт.**
- компонент `<NodeCard nodeId="..."/>` — читает `ui` из концепта `kind`, диспатчит виджеты по слотам;
- реестр виджетов (slug → React-компонент);
- 10–12 стартовых виджетов:
  - **Universal**: `Title`, `Description`, `Tags`, `WikidataLink`, `Timestamps`, `Breadcrumb`;
  - **Edge-based**: `LocationCrumb` (← `contains`), `ContainedItems` (→ `contains`), `LentToCard` (→ `lent_to`);
  - **Field-based**: `CategoryChip`, `BrandChip`, `QuantityDisplay`;
- декларация `ui` на концепте `kind_item` в seed;
- роут `/items/[id]` использует `<NodeCard>`.

**Валидирует.** Композиция из каталога работает; добавил виджет в `ui.header` — он появился без правки `<NodeCard>`.

**DoD.** Открыл `/items/перфоратор_makita_hr2630` — увидел карточку: брёд + название + категория, ниже описание, сбоку — где лежит (полка → шкаф → гараж), вкладка «Содержит» пустая, действия [одолжить].

**Усилие.** 2–3 дня.

### Фаза 4: `<NodeList>` с фильтрами

**Цель.** Список вещей с фильтрами и поиском, всё из каталога.

**Артефакт.**
- `<NodeList kind="item"/>` — читает `ui.list_columns` из концепта, рендерит таблицу;
- фильтры:
  - по **категории** (с обходом иерархии: выбрал «инструмент» — попадает и `power_tool`/`rotary_hammer`);
  - по **локации** (рекурсивно: выбрал «гараж» — попадает всё внутри);
  - по **бренду**;
  - по **тегам** (через `related_to`);
- полнотекстовый поиск по `name` (использует существующий BM25-индекс из `database.md`);
- пагинация / virtual scroll;
- сортировка по колонкам;
- кнопка «Добавить» (фаза 5).

**Валидирует.** Иерархия категорий не просто хранится — реально работает в UI-навигации.

**DoD.** На `/items` фильтр «садовый инвентарь» показывает и `garden_tool`-айтемы, и `seeds`, и `sapling`; фильтр «гараж» — все вещи в гараже включая вложенные шкафы.

**Усилие.** 2 дня.

### Фаза 5: `<NodeForm>` создание/редактирование

**Цель.** Декларативные формы из `ui.form_create` на концепте.

**Артефакт.**
- `<NodeForm kind="item" mode="create|edit"/>`;
- генерация полей формы из декларации в концепте;
- селекторы для рёбер: `LocationPicker` (где лежит → `contains`), `BrandSelect` (из vocab:domovoy_brand), `CategorySelect` (с автокомплитом из vocab:domovoy_category);
- валидация на стороне клиента + DB-`ASSERT`/`EVENT` (`fn::check_edge`) ловит нарушения каталога;
- загрузка фото (опц., в MinIO — но без OCR/процессинга в этой фазе);
- маршрут `/items/new` и `/items/[id]/edit`.

**Валидирует.** Та же декларация управляет и просмотром, и редактированием; добавление узла = `CREATE thing` + `RELATE` без хардкода.

**DoD.** Создал узел «дрель Bosch» через UI — она появилась в списке, попала в правильную локацию, имеет правильную категорию и бренд.

**Усилие.** 2–3 дня.

### Фаза 6: `<LocationTree>` — специализированный UX

**Цель.** Удобная навигация по дереву хранения.

**Артефакт.**
- маршрут `/storage` с деревом локаций (квартира → комната → шкаф → полка → ящик);
- клик по узлу дерева → справа фильтрованный `<NodeList>` всех вещей в этой локации (рекурсивно);
- drag-n-drop: перетащил вещь в другую локацию → ребро `contains` обновилось;
- breadcrumb путь от корня к выбранной локации.

**Валидирует.** Специализированный экран строится из тех же примитивов (`<NodeList>` + дерево как обычный обход графа), не создавая «зоопарк компонент». Архитектура выдерживает доменную специфику.

**DoD.** Открыл `/storage`, кликнул «гараж → верстак» — увидел все 30 инструментов на верстаке; перетащил отвёртку на полку — изменение сохранилось.

**Усилие.** 2 дня.

### Фаза 7: i18n переключатель

**Цель.** Прокатать симметричную многоязычность.

**Артефакт.**
- переключатель UI-языка в шапке (ru/en);
- все метки (`kind`-имена, категории, бренды, статусы, метки виджетов) приходят из `_i18n` концептов;
- контент-поля (`name`, `description`) остаются на оригинальном языке (правило про оригинал из обсуждения SKOS);
- индикатор «оригинал на ru» + кнопка «перевести» (без реального перевода — заглушка, реально появится в AI-вертикали).

**Валидирует.** Никакой «канонический язык» не зашит в код; UI равно дружелюбен к ru и en; принцип «идентификатор ≠ метка» работает.

**DoD.** Переключил язык — увидел «Refrigerator» вместо «Холодильник» в категории и «Hammer drill» в типе, но `name` вещи остался «Перфоратор Makita HR2630» (как ввёл).

**Усилие.** 1–2 дня.

### Фаза 8: Права и шаринг

**Цель.** Свобода-в-данных + жёсткость SurrealDB-PERMISSIONS по `access-control.md`.

**Артефакт.**
- `DEFINE ACCESS account ON DATABASE TYPE RECORD` (SIGNIN/SIGNUP);
- `DEFINE TABLE thing PERMISSIONS FOR select/update/delete WHERE ...` — обобщённый шаблон, читающий `can_access`;
- `DEFINE TABLE can_access PERMISSIONS` — кто вправе создавать/менять гранты;
- логин/регистрация в UI;
- `<AccessGrantsWidget>` на карточке узла: «у Папы view+edit, у Мамы view, у Сына нет»;
- модалка «поделиться» — добавляет `can_access` ребро;
- запросы получают **только то, что разрешено** для авторизованного пользователя.

**Валидирует.** Главный тезис проекта — свобода живёт в данных, enforcement один и читает граф. Член семьи без прав на «кошелёк папы» его не видит, без правки схемы.

**DoD.** Зарегистрированы 3 пользователя (папа/мама/сын). Папа видит весь инвентарь, мама не видит «инструменты в гараже» (т.к. не получила access), сын видит только «детские игрушки». Папа поделился полкой с инструментами → мама сразу её видит.

**Усилие.** 2–3 дня. Самая «концептуально весомая» фаза — здесь весь access-control оживает.

### Фаза 9: `lent_to` workflow

**Цель.** Прокатать действия-виджеты, меняющие граф.

**Артефакт.**
- виджет-действие «Одолжить» на карточке `item` → диалог «кому?» → создаёт `lent_to` ребро с датой;
- виджет-действие «Вернуть» → удаляет ребро, опц. лог в `log_entry`;
- список `/lent` — что сейчас одолжено и кому, с возрастом займа;
- (опционально, если делать сразу) `worker-scheduler` минимальный — раз в день шлёт уведомление, если что-то одолжено больше N дней.

**Валидирует.** Action-виджеты работают; user-workflow поверх графа без хардкода.

**DoD.** Папа одолжил соседу велосипедный насос → запись появилась в `/lent` → через 14 дней пришло уведомление.

**Усилие.** 1 день (без воркера), 2 дня (с воркером).

### Фаза 10: PWA + мобильный

**Цель.** Можно пользоваться с телефона стоя в магазине / в гараже.

**Артефакт.**
- адаптивная раскладка работает автоматически (через переписывание `header`/`body` в столбик на mobile-ширине — без отдельного `ui.mobile`);
- PWA-манифест, иконки, splash;
- service worker для офлайн-просмотра списков (что закешировано → доступно);
- камера для добавления фото к узлу;
- (опц.) сканер штрих-кода → запрос к Wikidata/GS1 → автозаполнение брэнда/категории.

**Валидирует.** Адаптив без отдельной разметки — заявленное преимущество универсальной композиции работает.

**DoD.** Открыл с телефона в гараже, добавил новую коробку с шурупами — фото с камеры, штрих-код прочитан, категория `fastener` подставилась автоматически.

**Усилие.** 2–3 дня.

## Что после инвентаря — почему ничего не придётся переделывать

Каждая следующая вертикаль — это **новые данные в каталоге + декларация `ui`**, без архитектурных изменений:

| Следующая вертикаль | Что добавится |
|---|---|
| **Документы и файлы** | `kind='document'/'file'/'chunk'`, виджеты `MimeIcon`/`PageList`/`OcrText`, MinIO-загрузчик |
| **Задачи / GTD** | `kind='task'`, виджеты `StatusChip`/`AssigneeAvatar`/`DueDate`/`Subtasks` (уже в плане ui-architecture) |
| **Питомцы / растения** | `kind='animal'/'plant'`, виджеты `LifecycleStage`/`CareSchedule`/`HealthLog` |
| **Медицина** | `kind='diagnosis'/'appointment'/'prescription'`, виджеты `IcdCode`/`AtcCode`/`DosageDisplay` |
| **IT-инфра** | `kind='device'/'service'/'incident'`, виджеты `OnlineIndicator`/`Hostname`/`CveLink`/`LogTimeline` |
| **AI-чаты** | `kind='chat'/'message'`, виджеты `MessageBubble`/`InputsPanel`/`ModelBadge`, интеграция Ollama |

Каждая — 3–7 новых виджетов + декларации + опц. воркер. **Ноль рефакторинга**.
