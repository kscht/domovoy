# Категоризация как процесс

Как в Домовом появляются категории — не как одно действие, а как **многомодальный процесс**, растянутый во времени, с разными ролями и разными источниками подсказок.

Связанные документы: [`kind-audit.md`](kind-audit.md) (стартовое состояние каталога), [`relation-typing.md`](relation-typing.md) (SKOS-словари и градация строгости), [`ui-architecture.md`](ui-architecture.md) (виджеты, которые это всё рендерят), [`roadmap-inventory.md`](roadmap-inventory.md) (фазы внедрения).

## Базовый сдвиг

В классических админ-системах категория — обязательное поле в форме создания. Один момент, одно действие, одна ответственность.

У нас иначе:

> **Категория — это опциональная метаданная, которая может появиться в один из шести разных моментов, любым из пяти разных способов, силами любого из трёх типов акторов.**

Это многомерный процесс, а не действие. Поэтому единого «UI категоризации» быть не может — есть **набор контекстов с разной стоимостью и разными аффордансами**.

И стартовое состояние каталога — **полностью пустое**. Дерево категорий вырастает из реальных кликов пользователя, а не из моих предположений о его жизни.

## 6 моментов, когда категория появляется или меняется

| # | Момент | Триггер | Стоимость UX |
|---|---|---|---|
| 1 | **Quick add без категории** | бросить вещь в систему, чтобы не забыть | **секунды** — только название + локация |
| 2 | **Considered add с категорией** | планомерно занести с полями | минуты, полная форма |
| 3 | **Inline triage** (разбор «без категории») | пошёл чистить хвост накопленных | минуты на группу из bulk-операции |
| 4 | **Subdivision** (выделить ребёнка) | заметил перенаселённость категории | минуты, мини-форма + bulk-переразметка |
| 5 | **Reclassification** (переместить) | передумал, нашёл лучшую категорию | секунды на айтем / минуты на bulk |
| 6 | **External enrichment** | сканер штрих-кода / Wikidata-look-up | секунды, автоматически предложение |

Главное: момент **(1) — отдельный полноценный режим**, не «недозаполненный (2)». Пустая категория — легитимное состояние, не баг.

## 5 источников подсказок

Когда категория всё-таки нужна, автокомплит ранжирует кандидатов из этих источников:

| Сила | Источник | Когда срабатывает |
|---|---|---|
| 🟢 высокая | **GTIN-скан → Wikidata** | сканер штрих-кода в мобильном PWA, точное совпадение по коду |
| 🟢 высокая (только разбор) | **`legacy_kind`** | при миграции seed: «было `крепёж` → создать `fastener`?» |
| 🟢 высокая (позже) | **AI vision на фото** | сделал снимок при добавлении (когда появится AI-воркер) |
| 🟡 средняя | **Поиск по имени** в существующих категориях + в Wikidata | напечатал «iPhone» → `phone` если есть, или предложить из Wikidata |
| 🟡 средняя | **Контекст локации** | «в аптечке» → `medicine`; «на верстаке» → `tool` |
| ⚪ fallback | **Ручной ввод** | сам печатаешь, ничего не угадали |

Все источники собираются в **один автокомплит-инпут**. Пользователь не выбирает «откуда искать» — он печатает или сканирует, ранжирование под капотом.

## 3 роли в социальном контракте

Категоризация имеет «социальный слой» — кто-что-кому разрешено. Три логические роли:

| Роль | Что делает | Кто в семье |
|---|---|---|
| **Контрибьютор** | добавляет айтемы, ставит/не ставит категории | кто угодно с доступом |
| **Сборщик** | разбирает «без категории», группирует, создаёт новые | обычно тот, кто завёл систему |
| **Куратор** | переименовывает, мержит, переродителит, удаляет | админ семьи (1–2 человека) |

Один человек может играть все три роли. Логическое разделение нужно для **градации прав** через `enforce` на `vocab:domovoy_category`:

| Режим | Кто создаёт | Кто курирует | Когда подходит |
|---|---|---|---|
| **open** | любой | никто | соло-юзер или маленькая семья |
| **soft** | любой, помечается «новая, непроверена» | админ периодически чистит | типичная семья 3–5 чел |
| **strict** | только админ; остальные через «запрос» | админ | если боишься хаоса (≥ 5–10 пользователей) |

Стартовая рекомендация: **soft**. Категория создаётся свободно, но виджет помечает её бейджем «new» в дереве, пока админ не подтвердил.

## Категория опциональна и отделима

Это самое важное архитектурное решение. В большинстве систем «айтем без категории» = ошибка / временное состояние. У нас — **легитимный режим**:

```surql
CREATE thing:`unknown_box_1` SET kind='item', name='Какая-то коробка с дачи';
-- category отсутствует. Айтем валиден.
```

Что это даёт:
- **быстрый импорт чего угодно** — фото + название → уже в системе, разбор потом;
- **разделение «зафиксировать существование» и «осмыслить, что это»** — два разных когнитивных акта;
- **легитимный inbox** «без категории» как отдельный режим UI.

## «Без категории» — это inbox-режим

UI трактует айтемы без `category` как «inbox задач на категоризацию». Отдельный экран:

```
┌─ Вещи без категории ─────────────────────── 1004 ──────┐
│ Группировать по: [legacy_kind ▼]                       │
│                                                          │
│ ▼ электроинструмент  (25)                  [☰ Bulk]    │
│   ☐ Перфоратор Makita HR2630                            │
│   ☐ Дрель Bosch GSR 12V                                 │
│   ☐ Шуруповёрт DeWalt DCD771                            │
│   ... ещё 22                                            │
│   [Выбрать все 25]  [→ Применить категорию...]         │
│                                                          │
│ ▶ хозтовар           (38)                              │
│ ▶ инструмент         (79)                              │
│ ▶ крепёж             (62)                              │
│ ▶ ... ещё 52 группы                                     │
└─────────────────────────────────────────────────────────┘
```

Клик «Применить категорию...» открывает inline-форму создания/выбора + bulk-перезапись `category` у всех выбранных айтемов в одной транзакции. **За один вечер ~50 legacy-групп разносятся в твоё дерево из 20–30 категорий.**

## Множественность и неоднозначность

«Велосипедный фонарь — это `lighting` или `bicycle_accessory`?»

| Случай | Как решается |
|---|---|
| Айтем подходит под две категории | основная одна (где чаще ищешь), вторая через `related_to → tag:bicycle` |
| Категория должна жить под двумя родителями | `concept` может иметь **несколько `part_of`-рёбер** — это норма (`agrochemistry` ↔ и `garden_supplies`, и `chemistry`) |
| Многомерная классификация (страна / опасность / органическое) | вообще не категория, а **теги** через `related_to {label:'тег'}` |
| Внешняя таксономическая глубина | поле `wikidata=Q...` на айтеме → бесплатная W-цепочка `subclass_of` |

`category` — это **главная навигационная метка**. Теги, Wikidata, иерархия — всё ортогонально.

## Inline создание категории: точка пересечения всех контекстов

Один UI-паттерн обслуживает 80% случаев. Это диалог, появляющийся в момент, когда пользователь набирает категорию, которой нет.

**Что видит пользователь:**

```
┌─ Новая вещь / Категория ────────────────────────────────┐
│ [теле|                                              ]    │
│  ┌─────────────────────────────────────────────────┐    │
│  │ 📁 Электроника (фильтрует 40 вещей)            │    │
│  │ ✨ + Создать «телефон»                          │    │
│  │      ↳ под «Электроника» (по контексту)        │    │
│  │      Wikidata: Q22645 «telephone» ✓             │    │
│  └─────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────┘
```

**Клик «+ Создать»:**

```
┌─ Новая категория ───────────────────────────────────────┐
│ Метка ru:    [Телефон                              ]    │
│ Метка en:    [Phone                                ]    │
│ Slug:        [phone                              ] 🔄    │
│              ↑ от англ. метки (можно сменить)          │
│ Wikidata:    Q22645 — telephone ✓                       │
│ Родитель:    [Электроника ▼]                            │
│                                                          │
│              [Отмена]    [Создать и применить]          │
└──────────────────────────────────────────────────────────┘
```

**Что технически:**

```surql
BEGIN TRANSACTION;
CREATE thing:`concept_phone` SET kind='concept', identifier='phone',
  wikidata='Q22645',
  _i18n={ ru:{label:'Телефон'}, en:{label:'Phone'} },
  created_by=$auth, created_at=time::now(),
  is_provisional=true;                         -- бейдж "new" в дереве до подтверждения куратором
RELATE thing:`concept_phone`->part_of->thing:`concept_electronics`;
UPDATE thing:`iphone_15_pro_maks` SET category='phone';
COMMIT TRANSACTION;
```

Это **один пользовательский акт**, под капотом — три операции в одной транзакции.

│   ▶ computer                                     (3)    │

Куратор раз в месяц проходится — мержит дубликаты, перерасполагает, подтверждает «new». Между визитами система работает.

## Качество категоризации — функциональное, не научное

Не нужно «правильно» категоризировать с точки зрения биологии/химии. Категория хороша, если она **помогает быстро найти / восстановить / решить задачу**. Метрики:

| Что | Норма | Что делать вне нормы |
|---|---|---|
| Размер группы | 5–50 айтемов | <5 → слить с родителем; >100 → подразбить |
| Частота использования фильтра | хотя бы раз в полгода | не использовалась — кандидат на удаление |
| Связь с workflow | категория = «что я с этим делаю» | если только описательная — может стать тегом |
| Дублирование с Wikidata | не повторяет 1-в-1 W-классификацию | если повторяет — оставь только Wikidata-якорь |

UI может **подсвечивать** аномалии (бейдж «80 айтемов, может пора подразбить?») — но не действовать автоматически. Решает человек.

## Эволюция дерева во времени

Категоризация — это **садоводство**, не архитектура. Дерево растёт постепенно:

| Этап | Что в дереве | Сколько категорий |
|---|---|---|
| День 1 | 0 категорий, всё «без категории» | 0 |
| Неделя 1 | 5–10 верхнеуровневых, плоско | ~7 |
| Месяц 1 | начали появляться поддеревья (`tool/{hand,power}`) | ~15 |
| Месяц 3 | два уровня глубины в нагруженных ветках | ~25 |
| Год | стабильно ~40 категорий, три уровня где надо | ~40 |

Каждая итерация = **данные**, не миграция, не релиз. Граф принимает любые изменения каталога в режиме реального времени.

## Когда подключится AI-помощник (позже)

Шестой источник подсказок (vision + clustering) появится в AI-вертикали:

- **Voice / AI vision на фото** → «это похоже на электроинструмент» — `power_tool` предложен;
- **Кластеризация по тексту имени** → «у тебя 25 айтемов в `electronics` с именем `iPhone`/`Xiaomi`/`Samsung` — выделить `phone`?»;
- **Внешнее обогащение из открытых датасетов** (food databases, hardware catalogs) — автоматическое обогащение полей и категорий.

Это **усиление**, а не замена. Человек остаётся в петле — система предлагает, человек подтверждает.

## Принципы дизайна

1. **Категоризация опциональна.** Айтем без категории — норма. «Без категории» — режим, не ошибка.
2. **Категория — главная навигационная метка.** Многомерность через теги, иерархия через `part_of`, внешняя глубина через `wikidata`.
3. **Создание — в потоке.** Inline в момент использования, не отдельным админ-ритуалом.
4. **Bulk важнее точечного.** Группа из 25 айтемов = один акт категоризации.
5. **Wikidata делает работу.** Не изобретай таксономию там, где она уже есть.
6. **Курирование — редкими волнами.** Не блокирует ежедневный поток.
7. **Стартуем с пустоты.** Каталог пуст, дерево вырастает из реальных кликов.
8. **Функциональное качество.** Категория полезна, если помогает в задаче, а не если научно правильна.

## Открытые вопросы

1. **`is_provisional` флаг для непроверенных категорий** — нужен ли как отдельное поле или достаточно `created_by != admin`?
2. **Лимит на глубину иерархии** — стоит ли ограничить (например, ≤ 4 уровня), чтобы не выродиться в Borges?
3. **Конфликт-разрешение** при создании двумя пользователями одинаковой категории одновременно — last-write-wins или явный merge prompt?
4. **AI-vision источник** — как ранжируется относительно человеческого ввода? Доверяем больше или меньше?
5. **Удаление категории с айтемами** — куда переезжают айтемы? В родительскую категорию, в `misc`, или блокируем удаление?
6. **«Запрос на категорию»** в strict-режиме — отдельная сущность (`kind='request'`) или просто `task` с особым typом?

## Резюме

> Категоризация в Домовом — это **процесс**, не действие. Она опциональна, многомодальна, растянута во времени и подразумевает разные роли. **Стартовая позиция — пустой каталог**, дерево вырастает из реальных кликов с помощью inline-создания и bulk-разбора. Wikidata автоматически снимает 80% работы при сканировании. Курирование происходит отдельными волнами, не блокируя ежедневный поток. Качество категории функциональное, не научное.

Один технический фундамент (`vocab:domovoy_category` + концепты + `part_of`-иерархия), много UI-паттернов под разные ситуации, градуируемая социальная политика.
