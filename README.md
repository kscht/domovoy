# Домовой

Единая self-hosted система управления жизнью на основе графа. Заменяет десятки разрозненных приложений — от списка покупок до IT-инфраструктуры — одной моделью данных.

## Идея

Всё есть **вещь**. Полка, мотоцикл, задача, рецепт, врач, жалоба, трекинг-номер, сервер, AI-чат, плейбук, дашборд — один тип узла. Смысл задаётся рёбрами между узлами.

```
Задача: ТО мотоцикла
  ├── assigned_to → Папа
  ├── part_of     → Подготовка к сезону
  ├── depends_on  → Купить масло и фильтры
  └── produces    → Запись о ТО (исторический факт)

Сервер: Proxmox-home
  ├── contains    → ВМ: Nextcloud
  ├── contains    → ВМ: Pi-hole
  ├── located_at  → Стойка в кладовке
  └── about       ← Инцидент: диск заполнен

AI-чат: Анализ инцидента
  ├── about       → Сервер: Proxmox-home
  ├── produces    → Задача: заменить диск
  └── inputs      → RAG: последние инциденты из графа
```

Это позволяет моделировать что угодно без новых таблиц и без переключения между приложениями.

| **Доставки** | Сложный трекинг, смена перевозчика, разбитые посылки |

| **Публикации** | Блог, сайт, Telegram-канал — граф как источник истины |

## Граф: 1 тип узла, 27 рёбер

```
contains      — где физически находится / содержит
part_of       — часть чего / подзадача / запуск шаблона
assigned_to   — кто отвечает
depends_on    — задача ждёт другой / сервис зависит от сервиса
about         — касается этой вещи или решения
filed_with    — подано в инстанцию
needs         — нужно купить (quantity, unit)
requires      — плановый расход шаблона
produces      — результат выполнения / публикация
used          — фактический расход
participant   — участник с ролью
located_at    — где происходит событие (started_at, ended_at)
lent_to       — отдано во временное пользование (since)
borrowed_from — взято у человека (since)
triggered_by  — активируется при событии / расписании
expert_in     — эксперт в области
answered      — попытка → вопрос (chosen, correct, points_earned)
promised_to   — кому дано обещание
represents    — цифровая копия → физический оригинал
can_access    — кто имеет доступ (permissions, approval)
related_to    — произвольная связь (label)
references    — ссылка на узел с режимом и снапшотом текста
identified_by — идентифицируется меткой / QR / штрихкодом
linked_to     — физический или логический сетевой линк (schemaless)
invalidates   — изменение узла инвалидирует кэш / сборку
supersedes    — версия заменяет предыдущую (порядок версий; текущая = голова цепочки)
derived_from  — получено из (происхождение/lineage: method, run, at)
```

## Стек

| Слой | Технология |
|------|-----------|
| База данных | [SurrealDB](https://surrealdb.com) — граф, schemaless, SurrealQL, LIVE SELECT |
| Фронтенд | Next.js 15 (App Router), TypeScript |
| UI | Tailwind CSS + shadcn/ui |
| Кэш | TanStack Query |
| Файлы | MinIO (S3-совместимый, self-hosted) |
| Воркеры | Stateless workers — Kubernetes / bare-metal |
| PWA | next-pwa — офлайн, push-уведомления |

## Архитектура

```
SurrealDB  (граф + очереди + LIVE SELECT)
    ↑↓
App Servers (Next.js, stateless, ×N)
Workers     (плейбуки, AI, HTTP, файлы, ×M каждый)
    ↓
MinIO       (файлы, медиа, бинарные данные)
Prometheus  (метрики и телеметрия, опционально)
```

Воркеры конкурируют за задачи через атомарный захват из SurrealDB-очереди. Масштабируются горизонтально без изменений схемы.
